# v2.5.2 詰み局面における並列探索異常の調査記録

## 1. 概要

ShogiHome 上で ShogiAI v2.4.3（先手）と v2.5.2（後手）を対局させた際、
後手に短い詰みがある局面で v2.5.2 が `-2000` を表示したにもかかわらず、
詰みを継続しない不自然な手を指した。

対象棋譜:

```text
/home/kuro/Downloads/20260719_160332_連続対局_10_30_ShogiAI_v2.4.3_ShogiAI_v2.5.2.kif
```

ShogiHome の表示は先手基準になっており、棋譜でも `後手勝勢` と併記されている。
したがって、後手番での `-2000` は v2.5.2 が後手必勝、内部的には
`+infinity` 相当と判断したことを意味する。

外部の将棋解析ツールでは、問題の局面は後手の7手詰めと判定されている。

この文書は長時間教師生成中に実施した静的調査の記録である。CPU負荷を避けるため、
問題局面の再探索、ビルド、動的計測はまだ行っていない。

## 2. 棋譜上の異常

特に重要な箇所は次の通り。

```text
76 ４七歩成
   評価値=-2000
   深さ=7
   ノード数=793420

77 ３五歩
   評価値=-387

78 ６九飛成
   評価値=-2000
   深さ=30
   ノード数=76610

79 同玉
   評価値=-86

80 ３七と
   評価値=-86
```

76手目と78手目では必勝値を表示しているが、読み筋が棋譜に保存されていない。
78手目はわずか76,610ノードで深さ30に到達している。その直後に評価が通常値へ
戻り、後手は詰みを継続していない。

この挙動は単なる評価関数の誤差では説明しにくい。探索結果の統合、詰み値、
または履歴依存値のTT利用に問題がある可能性が高い。

## 3. 静的調査で確定した問題

### 3.1 表示した評価値と実際の指し手が異なる可能性

`find_best_move_parallel` では、メインスレッドだけが `info` を出力する。
ワーカースレッドは `emit_info=false` で探索する。

```rust
let best_move = worker.find_best_move_with_root_offset(...);
(best_move, worker)
```

メインスレッドの探索終了後、ワーカーがメインより深い深さを完了していれば、
評価値やPVを比較せずにワーカーの指し手へ差し替える。

```rust
let mut selected_move = main_move;
let mut selected_depth = self.last_completed_depth;
for handle in handles {
    let (worker_move, worker) = handle.join().expect(...);
    if worker.last_completed_depth > selected_depth {
        selected_depth = worker.last_completed_depth;
        selected_move = worker_move;
    }
}
```

ワーカーから返される情報には、ルート評価値、PV、詰み手数、探索が完全に終了した
かどうかが含まれていない。

このため、ShogiHome に表示された `-2000` と、最終的な `bestmove 6九飛成` が
別スレッドの探索結果である可能性がある。これは設計上確実に発生し得る問題である。

### 3.2 ワーカー選択が探索深さだけに依存する

最終手の選択基準は `last_completed_depth` のみである。

- メインが浅い深さで詰みを発見していても、その情報は比較に使われない。
- ワーカーの評価値が低い、PVが空、境界値しか得られていない場合でも選ばれ得る。
- 同じ深さならメインを維持するが、時間切れのタイミングによって結果が変わる。
- 棋譜に記録される評価値と最終手の対応が保証されない。

並列探索が正しい手を返すためには、少なくとも各ワーカーから
`move, score, pv, completed_depth, bound, worker_id` を一体として返す必要がある。

### 3.3 詰みがすべて無限大で、詰み手数を比較できない

合法手がない局面は `-f32::INFINITY` として扱われる。

```rust
if moves.is_empty() {
    return Some((-f32::INFINITY, Vec::new()));
}
```

この方式では次を区別できない。

- 1手詰みと長い詰み
- 正しく証明された詰みと、別経路からTTへ混入した無限大
- 詰まされる全候補のうち、最も詰みを長く延ばす手

ルートの更新条件は厳密な `>` であるため、一度 `+infinity` になった候補は、
後から別の正しい詰み手が `+infinity` になっても置き換えられない。
探索順序がそのまま指し手選択へ影響する。

### 3.4 `alpha=+infinity` 後に不正な探索窓が生成される

ルートPVSでは、`alpha` が有限の場合だけ null-window search を使い、それ以外は
通常探索へ進む。

```rust
if current_best_move_for_depth.is_some() && alpha.is_finite() {
    ...
} else {
    self.alpha_beta_search(position, depth - 1, -beta, -alpha)
}
```

最初の候補で `alpha=+infinity` になると、次の候補へ渡す窓は次になる。

```text
[-beta, -alpha] = [-infinity, -infinity]
```

これは有効な alpha-beta window ではない。静止探索では任意の評価値が
`stand_pat >= -infinity` を満たすため、即座に `-infinity` を返し、ルート側で
`+infinity` と解釈され得る。

結果として、本物の詰みを1つ発見した後、未検証の他候補まで必勝値として扱う
危険がある。

### 3.5 最善手・PVがない深さも完了扱いになる

`current_best_move_for_depth` が `None` でも、探索が時間切れでなければ
`last_completed_depth` を更新する。

```rust
if !search_interrupted {
    if let Some(current_best_move) = current_best_move_for_depth {
        best_move = Some(current_best_move);
        previous_eval = Some(best_eval_for_depth);
    }
    self.last_completed_depth = depth;
}
```

これにより、PVを構築できなかったワーカーが「より深い探索を完了した」と見なされ、
並列探索の最終手として選ばれる可能性がある。

### 3.6 千日手判定が連続王手を完全には検証していない

現在の千日手判定は、同一手番の同一局面が4回出現し、その局面が王手されている場合に
`PerpetualCheckLoss` を返す。

```rust
if count >= 4 {
    if position.in_check() {
        SennichiteStatus::PerpetualCheckLoss
    } else {
        SennichiteStatus::Draw
    }
}
```

しかし、連続王手の反則判定には、繰り返し区間における同じ側の全着手が王手だった
ことの確認が必要である。現在は各着手の王手情報と王手した側を履歴に保存していない。

深い探索中に通常の反復を連続王手と誤認すると、`+/-infinity` が発生する。

### 3.7 履歴依存の値が共有TTへ伝播する

千日手や連続王手の結果は、その局面へ到達した履歴に依存する。一方、TTキーは盤面、
持ち駒、手番のハッシュだけで、探索履歴を含まない。

反復を検出したノード自体は直接TTへ保存されなくても、その結果を受け取った祖先ノード
の `+/-infinity` はTTへ保存される。そのエントリを異なる履歴のワーカーが共有TTから
読むと、別経路でも詰みまたは必敗と誤認する可能性がある。

v2.5.2は既定の `Threads=0` で全CPUを使い、全ワーカーが同じTTを共有するため、
単一スレッドより影響範囲が広い。

### 3.8 USIの実戦局面履歴を千日手検出器へ渡していない

`position ... moves ...` の処理では盤面へ着手を適用するが、その途中局面を
`sennichite_detector` へ記録していない。

そのため、実戦ですでに発生している反復回数を探索が認識できない。今回の直接原因
とは限らないが、千日手処理として別途修正が必要である。

## 4. 現時点の原因仮説

優先度順に次を想定する。

### 仮説A: メインの詰み評価とワーカーの指し手が混在した

1. メインスレッドが正しい7手詰めを発見する。
2. メインが必勝値を `info` として出力する。
3. 別ワーカーがメインより深い深さを完了する。
4. 評価値を比較せず、ワーカーの指し手を `bestmove` にする。
5. ShogiHomeにはメインの `-2000` とワーカーの指し手が同じ手の情報として保存される。

現行コードだけで発生が保証され得るため、最有力である。

### 仮説B: 無限大探索窓によって詰まない手まで必勝になった

1. いずれかの候補で本物または誤った `+infinity` を得る。
2. `alpha=+infinity` になる。
3. 残り候補を `[-infinity, -infinity]` で探索する。
4. 未検証の候補も `+infinity` になる。
5. 最初に必勝値となった手が探索順序だけで残る。

### 仮説C: 千日手由来の無限大が共有TTを汚染した

深さ30へ少ないノード数で到達している点は、詰み値や反復値がTTを通して急速に
伝播した挙動とも整合する。ただし、実際にこの棋譜で発生したことを確定するには
再現時の計測が必要である。

## 5. 現在の長時間学習への影響

現在実行中の `scripts/train_halfkp64_large.sh` の教師生成器は、候補手ごとに
新しい `ShogiAI` を作り、`alpha_beta_search` を直接呼んでいる。
`find_best_move_parallel` は使用していない。

したがって、次の問題は現在の教師生成へ直接影響しない。

- メインとワーカーの表示・指し手不一致
- 深さだけによるワーカー選択
- 共有TTを使う並列ルート探索

教師探索は深さ4、難局面でも深さ5であり、4回反復に到達する可能性も低い。
無限大と千日手処理は共通コードに存在するが、今回の長時間処理を中止すべき根拠は
現時点ではない。

## 7. 改善案

### 7.1 まずメインスレッドの手だけを採用する

最も安全で変更範囲が小さい暫定修正。

- ワーカーは共有TTを温める補助に限定する。
- 最終的な `bestmove` は、表示した評価値とPVを持つメインスレッドから返す。
- これによりKIFの評価値、PV、指し手が必ず一致する。

強さへの影響を測定した後、ワーカー結果の正式な統合を設計する。

### 7.2 ルート探索結果を構造体として扱う

```rust
struct RootSearchResult {
    best_move: Move,
    score: Score,
    pv: Vec<Move>,
    completed_depth: u8,
    bound: NodeType,
    worker_id: usize,
    complete: bool,
}
```

メインと全ワーカーがこの構造体を返す。選択した結果を、そのままUSI `info` と
`bestmove` の両方へ使用する。

### 7.3 有限のmate scoreと詰み距離を導入する

例:

```text
MATE_SCORE = 1_000_000
詰ませる側: MATE_SCORE - ply
詰まされる側: -MATE_SCORE + ply
```

これにより次が可能になる。

- 最短の詰みを選ぶ
- 詰まされる側は最長の手順を選ぶ
- 通常評価値と詰み値を明確に分離する
- `alpha` と `beta` を有限値に維持する

USI出力も可能なら `score cp 2000` ではなく `score mate N` を使用する。

### 7.4 無限大またはmate boundaryでPVSを行わない

- `alpha >= MATE_THRESHOLD` の場合は残りのルート候補を探索しない、または正しい
  mate-distance windowを使う。
- `alpha < beta` を必ずassertまたは検査する。
- `[-infinity, -infinity]` の探索呼び出しを禁止する。

### 7.5 完全な結果だけを完了深さとして記録する

`current_best_move_for_depth` と有効なPVが存在する場合だけ
`last_completed_depth` を更新する。

時間切れ、空PV、境界値のみ、候補未選択の場合は、その深さをワーカー比較に使わない。

### 7.6 TTでmate scoreを正規化する

mate scoreをTTへ保存・取得する際は、ルートからのplyを加減して正規化する。
これにより異なる探索plyで同じ詰み距離を誤解しない。

反復判定を含むサブツリーについては、少なくともExact値としてTTへ保存しない。
必要なら探索結果に `history_dependent` フラグを追加し、祖先へ伝播させる。

### 7.7 連続王手の履歴を正しく保存する

局面ハッシュだけでなく、各着手について次を記録する。

- 着手した側
- その着手が王手だったか
- 同一局面の繰り返し区間

4回反復区間において、同じ側の全着手が王手の場合だけ連続王手の反則負けとする。

### 7.8 実戦履歴をワーカーへ引き継ぐ

USI `position ... moves ...` を処理するときに各局面を履歴へ記録する。
並列ワーカー生成時には、メインの千日手履歴を複製する。

## 8. 必要な回帰テスト

1. 対象棋譜の7手詰み局面で正しい詰み手を選ぶ。
2. `Threads=1` と `Threads=0` で詰み手と評価値が一致する。
3. 出力PVの初手と `bestmove` が常に一致する。
4. メインより深いが低評価のワーカーを採用しない。
5. 全候補が詰み勝ちの場合、最短詰みを選ぶ。
6. 全候補が詰み負けの場合、最長手数の手を選ぶ。
7. `alpha < beta` が全探索呼び出しで成立する。
8. 通常の千日手と連続王手の千日手を区別する。
9. 異なる反復履歴から同じ盤面へ到達してもTT値が混入しない。
10. 固定局面を並列で繰り返しても指し手とPVが安定する。

## 9. 暫定運用

修正版が完成するまでは、対局時に次を推奨する。

```text
Threads=1
```

これにより少なくとも次を回避できる。

- 表示したメイン評価と別ワーカーの指し手が混ざる問題
- 共有TTを介したワーカー間の履歴依存値の伝播

単一スレッドでもmate-distance、無限大window、千日手、ローカルTTの問題は残るため、
恒久対策ではない。

## 10. 調査時点の判断

- 評価関数やHalfKP重みが直接原因である可能性は低い。
- 探索が必勝値を出した後の指し手選択と並列統合に明確な設計不備がある。
- `+/-infinity` を通常の探索値として使用していることが複数の不具合を連鎖させている。
- 共有TTと不完全な千日手判定が、誤った必勝値の発生源になり得る。
- 最優先修正は「メインの手だけを返す」「有限mate score」「有効な結果だけを完了扱い」
  の3点である。

## 11. 動的再現による原因確定と修正

2026-07-19に長時間学習終了後、78手目直前の局面を棋譜から復元して動的確認を行った。

修正前の結果:

```text
Threads=1:
info depth 3 score cp 2000 ... pv S*5g
bestmove S*5g

Threads=0:
info depth 3 score cp 2000 ... pv 3i6i+
...
bestmove 3i6i+
```

`Threads=0` は実戦棋譜と同じ誤った `3i6i+` を再現した。この実行では表示PVと
`bestmove` が一致していたため、今回の直接原因は単なる最終ワーカー差し替えではない。

確定した直接原因は次の連鎖である。

1. 詰みを `+/-infinity` で表現していた。
2. 1候補で詰みを見つけた後、ルートの `alpha` が `+infinity` になった。
3. 後続候補が `[-infinity, -infinity]` という不正な窓で探索された。
4. 未検証候補が必勝境界値として扱われ、その値が共有TTを通じてメイン探索にも戻った。
5. 探索順序上先に現れた `3i6i+` が詰み手と誤認された。

また、深さだけで別ワーカーの手を採用する問題も独立した確定バグであり、同時に修正した。

実装した修正:

- 詰みを有限値 `MATE_SCORE - ply` として表現する。
- TT保存時と取得時に詰み距離をルートplyから正規化する。
- 反復由来の勝敗値を詰み値と分離し、反復値と引き分け値をTTへ保存しない。
- すべてのalpha-beta・静止探索呼び出しで `alpha < beta` を実行時検査する。
- 最善手とPVが得られた深さだけを完了深さとして記録する。
- 並列ワーカーは共有TTを温める補助に限定し、最終手はinfoを出したメイン探索から返す。

修正後の同一局面:

```text
Threads=0:
info depth 3 score cp 2000 ... pv S*5g 6h6g 5g5h 6g6h
info depth 5 score cp 2000 ... pv S*5g 6h6g 5g5h 6g6h 3i6i+
bestmove S*5g
```

固定局面を単一スレッドと4スレッドで比較し、並列探索を8回繰り返して常に `S*5g`
を返す回帰テストを追加した。mate scoreのTT正規化も別の単体テストで確認している。

反復値のTT混入は防止したが、千日手履歴の完全な連続王手判定とUSI実戦履歴の
引き継ぎは、今回の短い詰み局面では発生しておらず直接原因ではない。これらは別の
履歴依存探索修正として残る。
