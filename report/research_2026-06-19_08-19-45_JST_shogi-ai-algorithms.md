# 将棋AIアルゴリズム調査と本プロジェクトへの適用方針

- 作成日時: 2026-06-19 08:19:45 JST
- 対象プロジェクト: `/home/kuro/shogi_rust`
- 現在の開発ブランチ: `improve-self-play-learning`
- 目的: 世の中の将棋AIアルゴリズムを多角的に調査し、本プロジェクトに取り入れやすい内容、取り入れにくい内容、今後の実装・検証優先順位を整理する。

## 1. 現状認識

本プロジェクトは現時点で、以下の特徴を持つ古典的な探索型将棋AIである。

- 評価関数は KPP + 駒得を中心とする疎な線形評価。
- 探索は AlphaBeta / Negamax 系。
- 置換表、反復深化、静止探索、指し手順序付けを持つ。
- v2.1.0 の教師あり学習済み重みが強さの基準線になっている。
- 直近の改良では、王手千日手処理、USI stop 対応、終局理由記録、王手手の順序付け、SEE による捕獲手順序付けを導入した。

直近の代表ベンチでは、現行HEADは v2.1.0 baseline に対して以下の結果を出している。

- seed 1401 / 20局: 16勝4敗
- seed 1701 / 10局: 9勝1敗
- seed 2201 / 40局: 30勝8敗2分、総合勝率 77.5%

このため、現時点では「v2.1.0 を大きく上回る」方向には進めている。ただし、100局以上の安定検証、負け棋譜の分類、USI時間管理を含む実戦条件での検証はまだ不足している。

## 2. 調査した主要系統

### 2.1 Bonanza / KPP / MMTO 系

Bonanza は、KPP/KPPT のような玉位置と駒配置の組み合わせ特徴を大規模に学習することで、将棋AIの評価関数学習を大きく前進させた系統である。

JAIR の Hoki/Kaneko 論文では、実用的な AlphaBeta 探索プログラムの評価関数を Minimax Tree Optimization (MMTO) により学習する方法が説明されている。論文の要点は以下である。

- 評価関数は線形・非線形の重み付き特徴として扱える。
- 探索結果が専門家棋譜の指し手と一致するように目的関数を設計する。
- 4000万以上のパラメータを少数のハイパーパラメータで調整する。
- Bonanza の 2013年世界コンピュータ将棋選手権優勝に大きく寄与した。

本プロジェクトとの相性:

- 非常に高い。
- 既存のKPP評価と教師あり学習コードに最も近い。
- AlphaZero型より低コストで、現在のRustコードに段階的に入れられる。

取り入れやすい要素:

- 棋譜の正解手と探索上の候補手を比較する学習。
- 兄弟手比較による重み更新。
- 探索結果を教師信号にする蒸留。
- L2正則化、学習率管理、検証ゲート。

取り入れにくい要素:

- 本格MMTOの完全再現。
- 数千万パラメータを高速に回す学習基盤。
- 大規模棋譜・教師評価の整備。

優先度:

- 高い。
- ただし、いきなりMMTO完全実装ではなく、現在の `kpp_learn` / `kpp_self_learn` を「壊れない教師あり再学習」と「探索蒸留」に寄せるのが現実的。

### 2.2 やねうら王型: AlphaBeta + 高度探索 + 高速評価

やねうら王は、現在の公開将棋AIの中心的存在の一つである。GitHub README によると、USI、入玉宣言勝ち、トライルール、ponder、MultiPV、秒読み・フィッシャー等の時間制御、超並列探索、定跡DB、巨大置換表、KPPT/KPP_KKPT/NNUE/SFNN評価に対応している。

本プロジェクトとの相性:

- 高い。
- 現在のエンジンもAlphaBeta系なので、探索・時間管理・定跡・終局規則は直接参考にできる。

取り入れやすい要素:

- USI時間管理の改善。
- go depth / movetime / btime / wtime / byoyomi / infinite / stop の安定対応。
- 指し手順序付けの強化。
- 置換表の保存・参照ポリシー改善。
- 王手・王手回避・捕獲・成り・SEE の順序付け。
- 定跡DBまたは簡易序盤ブック。
- 詰み探索。
- ベンチマーク・終局理由・局面別勝敗分析。

取り入れにくい要素:

- 256スレッド級の超並列探索。
- Stockfish由来の枝刈りをすべて安全に移植すること。
- 高度な時間制御、ponder、MultiPV、入玉規則まで一気に実装すること。

優先度:

- 最高。
- 今のコードベースで最も短期的な強さ向上に直結する。

### 2.3 NNUE

NNUE は、将棋向けに考案された Efficiently Updatable Neural-Network-based Evaluation Function である。元実装はやねうら王をベースに、評価関数をニューラルネットワーク型に置き換える形で実装されている。

Stockfish NNUE のドキュメントでは、NNUE の基本原理として以下が説明されている。

- 入力が疎であること。
- 連続する局面間で入力変化が小さいこと。
- 低精度整数推論に向いた単純なネットワークであること。
- 差分更新により、通常の深層ネットワークよりCPU探索に向くこと。

本プロジェクトとの相性:

- 中長期では非常に高い。
- 現在のKPP特徴は、NNUEのHalfKP/King-Piece系特徴と思想が近い。
- ただし、学習器・推論器・重み形式・差分更新が必要になるため、短期改良より重い。

取り入れやすい要素:

- まずは小型NNUEの非差分推論をRustに実装する。
- KPP特徴抽出を流用し、入力を疎ベクトルとして扱う。
- Python/PyTorchで小型ネットを学習し、Rust形式にエクスポートする。
- 既存KPP評価とNNUE評価を切り替え可能にする。

取り入れにくい要素:

- 強豪級NNUEの完全実装。
- SIMD最適化、int8/int16量子化、差分Accumulator。
- 数千万局面規模の学習。
- 教師評価生成パイプライン。

優先度:

- 中期の本命。
- まずは現在のAlphaBeta探索を安定強化し、ベンチ基盤を固めてから着手するべき。

### 2.4 dlshogi / 深層Policy-Value + MCTS

dlshogi は、AlphaGo/AlphaZeroの手法を参考にした深層学習系将棋AIである。リポジトリ構成には、ニューラルネットワーク学習、MCTS自己対局、USIエンジン、ONNX Runtime版などが含まれている。

本プロジェクトとの相性:

- 研究対象としては高いが、現在の実装への短期適用は低い。
- MCTS、Policy/Valueネットワーク、GPU推論、学習基盤が必要になる。

取り入れやすい要素:

- Policyネットワークだけを指し手順序付けに使う。
- 既存探索で生成したPVや最善手を教師にして、軽量Policyを学習する。
- 終盤・戦型・駒配置のパターン認識を補助評価として使う。

取り入れにくい要素:

- フルMCTSエンジンへの移行。
- GPU前提の高速推論。
- AlphaZero型の自己対局学習基盤。
- 深層モデルを本番探索に入れたうえでNPS低下を補うこと。

優先度:

- 低から中。
- 現時点では「将来の別系統」と位置づける。まずはAlphaBeta+NNUE路線を優先する。

### 2.5 AlphaZero / AobaZero 型

AlphaZero は、深層Policy/ValueネットワークとMCTSを自己対局で学習する方式である。DeepMindの解説では、AlphaZeroは手作り評価を深層ネットワークと汎用アルゴリズムで置き換え、自己対局で学習し、MCTSで探索を導くとされている。

AobaZero は、AlphaZero将棋実験の再現を目指した公開プロジェクトである。READMEでは「非常に大規模な実験で、再現は容易ではない」と説明されている。GitHub issueでは、2019年3月から2年2か月で3980万局を生成したこと、AlphaZeroとの差分推定、elmoに対する勝率などが報告されている。

本プロジェクトとの相性:

- 長期研究としては有益。
- ただし、現在のCPU中心・Rust探索エンジンを強くする目的には遠回り。

取り入れやすい要素:

- 自己対局データを検証付きで蓄積する思想。
- 学習済みモデル候補を旧モデルと対戦させて採否判定するゲート方式。
- 勝敗だけでなく、局面価値や指し手分布を学習する考え方。

取り入れにくい要素:

- フルAlphaZero再現。
- 数千万局以上の自己対局。
- GPU/TPU級の計算資源。
- MCTS + 深層推論を既存AlphaBeta探索に置き換えること。

優先度:

- 現時点では低い。
- 自己対局学習を行う場合も、AlphaZero型ではなく、既存AlphaBeta探索の評価関数を壊さない形で進める。

## 3. 本プロジェクトに取り入れるべき内容

### 3.1 短期で取り入れるべき内容

#### USI時間管理の完成

理由:

- 実戦投入・外部GUI・ベンチの前提条件である。
- 時間切れや `stop` 無視は強さ以前の問題になる。
- 既に実装とサブエージェント検証が進んでおり、コミット可能な状態に近い。

実装対象:

- `go depth`
- `go movetime`
- `go infinite` + `stop`
- `go btime wtime byoyomi`
- bestmove後のstdout flush
- 探索スレッド停止フラグ

#### ベンチマーク基盤の強化

理由:

- 現在は10局・20局・40局の結果に依存している。
- 1つの実験が本当に強いか、乱数・先後・開始局面の偏りで勝っただけかを判断しにくい。

実装・運用対象:

- 100局ベンチの標準化。
- seed固定。
- 開始局面のペアリング。
- 先後入れ替え。
- 終局理由の集計。
- 負け棋譜の自動抽出。
- `new / baseline / draw` の信頼区間表示。

#### 負け棋譜の分類

理由:

- 次の改善対象を推測で選ぶと、これまでのように多くの実験がノイズで終わる。
- 現行版の負け方を見れば、探索・評価・序盤・終盤のどこを直すべきかが分かる。

分類軸:

- 詰み逃し。
- 王手回避ミス。
- 駒損。
- 捕獲順序の失敗。
- 入玉対応。
- 千日手。
- 序盤不利。
- 時間切れ・stop不備。

#### 詰み探索・王手周り

理由:

- 将棋では終盤の一手ミスが即敗北になる。
- 現行エンジンのような浅め探索では、詰み探索の追加が効きやすい。

候補:

- rootでの即詰み探索。
- 王手生成専用関数の利用。
- 王手回避局面の指し手順序改善。
- 連続王手の深さ延長。
- 深さ・局面数上限付きの軽量詰み探索。

#### SEEの精密化

理由:

- 現在のSEE利用は指し手順序付けとして有効だった。
- さらに捕獲の安全性を改善すれば、探索の枝刈り精度が上がる可能性がある。

候補:

- 単純な被害/加害評価から、交換シーケンスを読むSEEへ拡張。
- 駒打ちを含む将棋特有の再捕獲可能性を近似。
- 駒価値だけでなく成り・王手・玉近傍を補正。

### 3.2 中期で取り入れるべき内容

#### 探索蒸留型の再学習

理由:

- 壊れた自己対局学習をそのまま回すのではなく、強い現行探索を教師として使う方が安全。
- v2.1.0重みを破壊せず、検証ゲートを通った重みだけ採用できる。

方針:

- 現行エンジンで局面ごとのPV、最善手、評価値を生成する。
- KPP+materialモデルを、その評価値または最善手に近づける。
- 学習後は必ず v2.1.0 baseline と現行固定版に対して対戦評価する。
- 50局未満の良結果では採用しない。

#### 小型NNUEプロトタイプ

理由:

- 最先端に近づくには、最終的に評価関数の表現力を上げる必要がある。
- KPPの線形評価だけでは局面の非線形な関係を表現しにくい。
- NNUEは将棋由来でAlphaBeta探索と相性が良い。

段階:

1. 既存KPP特徴を入力とした小型MLPをPythonで学習する。
2. Rustでfloat推論を実装する。
3. 既存SparseModelと切り替え可能にする。
4. ベンチで効果を見る。
5. 効果があればint16/int8量子化とAccumulator差分更新へ進む。

#### 序盤ブック

理由:

- 浅い探索では序盤で不利になりやすい。
- taya36やCSA棋譜から頻出・高勝率の序盤を抽出できる。

候補:

- SFEN prefix から合法手頻度を集計。
- 最初の数手だけブック利用。
- ブック手にランダム性を持たせる。
- ベンチではブックあり/なしを分けて評価。

### 3.3 長期で検討する内容

#### 本格NNUE

必要なもの:

- 大量局面生成。
- 教師評価生成。
- PyTorch学習器。
- Rust推論器。
- 量子化。
- SIMD最適化。
- 差分Accumulator。
- 重みファイル互換設計。

期待:

- 評価関数の質が大幅に上がる可能性がある。
- AlphaBeta探索を維持したまま強さを伸ばせる。

リスク:

- 学習基盤が重い。
- 非差分float推論ではNPS低下が大きい可能性。
- 十分なデータなしではKPP線形評価より弱くなる可能性。

#### 深層Policy/Value + MCTS

必要なもの:

- GPU推論。
- MCTS実装。
- Policy/Valueネットワーク。
- 自己対局生成。
- モデル選抜ゲート。
- 数千万局規模の訓練。

期待:

- AlphaZero型の柔軟な局面理解。
- 人間棋譜に縛られない新しい指し方。

リスク:

- 計算資源が圧倒的に足りない可能性。
- 現在のコード資産を活かしにくい。
- 短期のv2.1.0圧倒には遠回り。

## 4. 取り入れることが難しい内容

### 4.1 フルAlphaZero再現

難しい理由:

- AobaZeroでも3980万局を2年以上かけて生成している。
- GPU/TPU級の推論資源が必要。
- MCTSと深層ネットワークを中心に設計し直す必要がある。
- 現在のAlphaBeta/KPPエンジンとは設計思想が大きく異なる。

判断:

- 今は実装優先度を下げる。
- 自己対局学習の考え方だけを、小規模・検証ゲート付きで取り入れる。

### 4.2 強豪級NNUEの一気通貫実装

難しい理由:

- 学習データ、教師評価、量子化、差分更新、SIMDが必要。
- どれか一つ欠けると、NPS低下や評価品質不足で弱くなる。

判断:

- まず小型NNUEで効果測定する。
- 既存KPP評価とのハイブリッドまたは切り替えにする。

### 4.3 大規模並列探索

難しい理由:

- 並列AlphaBetaは実装難度が高い。
- 共有置換表、split point、探索停止、再現性、ロック競合が問題になる。
- 単スレッドの探索品質が固まっていない段階では、バグの切り分けが難しい。

判断:

- まず単スレッド性能と検証基盤を優先する。
- 並列化はUSI実戦対応と評価関数高速化の後でよい。

## 5. 実装・検証の推奨優先順位

### P0: USI時間管理の確定

理由:

- 外部GUI・実戦・ベンチの基盤。
- 既に実装とサブエージェント検証が完了に近い。

完了条件:

- `cargo test --all-targets` 通過。
- `go depth 1` がbestmoveを返す。
- `go movetime 100` がbestmoveを返す。
- `go infinite` + `stop` がbestmoveを返す。
- `go btime/wtime/byoyomi` がbestmoveを返す。
- 変更をコミットする。

### P1: 100局ベンチの標準化

理由:

- 40局で77.5%は有望だが、採否判断にはまだ揺らぎがある。
- 今後の実験を自律的に進めるには、強さ判定の自動化が必要。

完了条件:

- 100局を同一コマンドで実行可能。
- 終局理由と先後別勝率が出る。
- v2.1.0 baseline への勝率が安定して70%以上。

### P2: 負け棋譜分類と改善対象の確定

理由:

- 改良対象を探索の勘で選ぶ段階は終わりにするべき。
- 負けの主因が分かれば、次の実装が絞れる。

完了条件:

- 直近100局の敗局を分類。
- 主因上位2つを特定。
- 次の実装対象を1つに絞る。

### P3: 詰み探索・王手回避・SEE精密化

理由:

- 現行の強化方向と整合している。
- 導入コストがNNUEより低く、勝敗に直結しやすい。

完了条件:

- 各実験は最低20局、候補採用は最低40局以上。
- 現行固定版に対して勝率が悪化しない。
- v2.1.0 baselineに対する優位を維持する。

### P4: 探索蒸留型の再学習

理由:

- 破壊的な自己対局学習を避けられる。
- 現在の強い探索を教師として評価関数に知識を戻せる。

完了条件:

- 学習重みを別ファイル保存。
- baseline・現行固定版の両方に対戦評価。
- 検証ゲートを通った重みだけ採用。

### P5: 小型NNUEプロトタイプ

理由:

- 長期的な伸びしろが最も大きい。
- ただし検証基盤がないと失敗時に原因が分からない。

完了条件:

- Rustで推論可能。
- 既存評価と切り替え可能。
- 小規模ベンチでKPP単体を上回る兆候がある。

## 6. 当面の判断

現時点では、以下の順で進めるのが最も合理的である。

1. USI時間管理をコミットする。
2. 現行HEADを v2.1.0 baseline に対して100局評価する。
3. 敗局を分類する。
4. 詰み探索・王手回避・SEE精密化のうち、敗因に最も近いものを実装する。
5. それでも評価関数の限界が見えたら、探索蒸留または小型NNUEへ進む。

大規模自己対局を盲目的に回すのは避けるべきである。過去バージョンで自己対局学習が評価関数を壊した経緯があるため、今後の学習は必ず以下の条件を満たす必要がある。

- 学習前重みを保存する。
- 学習後重みは別名保存する。
- baseline対戦で採否判定する。
- 現行固定版にも対戦させる。
- 短い10局ベンチだけで採用しない。
- 悪化した重みは即座に破棄する。
